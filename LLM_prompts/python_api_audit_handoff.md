# Python API Argument Audit Handoff

This note summarizes findings from reviewing `mismatch_report`, which was generated by
`clubb_python_api/tests/argument_list_enforcer/argument_contract_audit.py`.

## Current Report Shape

`mismatch_report` currently reports:

- `PUBLIC_API_SIGNATURE_MISMATCH: 109`
- `MISSING_FORTRAN_SOURCE_MAPPING: 24`
- `PYF_WRAPPER_SIGNATURE_MISMATCH: 1`

The report is useful, but not every entry should be treated as a real public API defect
yet. The current audit still conflates three different contracts:

1. Public Python API signature vs raw source Fortran routine signature.
2. Public Python API signature vs f2py wrapper signature.
3. f2py wrapper call into the raw source Fortran routine.

Those contracts need separate checks because some wrappers intentionally supply source
Fortran arguments from stored module state, constants, or grid objects.

## Findings That Look Real

Many `PUBLIC_API_SIGNATURE_MISMATCH` entries are likely real under the desired strict
policy that public `clubb_api` routines match source Fortran `intent(in)` and
`intent(inout)` arguments in content and ordering.

Common real-looking patterns:

- Python puts `gr` first while Fortran has dimensions first and `gr` later.
- Python omits `stats` even though source Fortran treats it as `intent(inout)`.
- Python omits `err_info` for routines where source Fortran treats it as part of the
  non-output contract.
- Python includes dimension/helper arguments that do not appear in the raw source
  routine.
- Python and Fortran use the same arguments but in a different order.

The single `PYF_WRAPPER_SIGNATURE_MISMATCH` also looks real:

- `parameters_tunable.py::init_clubb_params -> f2py_init_clubb_params`
- Wrapper has `nparams`; expected pyf/source-side name is `nparams_in`.

## Findings That Are Audit Noise Or Model Gaps

### G-unit Source Mapping

`advance_helper.py::smooth_min -> f2py_smooth_min_scalar_array` is mapped to:

- `src/G_unit_test_types/smooth_min_max_tests.F90::smooth_min_max_tests_driver`

That is wrong. The audit should not derive public API contracts from G-unit test
sources. The current skip logic likely misses paths such as `G_unit_test_types`.

Recommendation:

- Exclude any `src` path part that starts with `G_unit_test`, not only exact matches
  like `G_unit_test`.

### Elemental Subroutines

The report lists these as `MISSING_FORTRAN_SOURCE_MAPPING`:

- `new_hybrid_pdf.py::calculate_w_params -> f2py_calculate_w_params`
- `new_hybrid_pdf.py::calculate_responder_params -> f2py_calculate_responder_params`
- `new_hybrid_pdf.py::calc_coefs_wpxp2_semiimpl -> f2py_calc_coefs_wpxp2_semiimpl`

These are not actually missing. The f2py wrapper calls the source routines inside
loops, and the source routines exist in `src/CLUBB_core/new_hybrid_pdf.F90`.
The likely parser bug is that the source routines are declared as `elemental subroutine`,
while the parser recognizes plain `subroutine` more reliably.

Recommendation:

- Update Fortran routine parsing to accept prefixes before `subroutine` and `function`,
  including at least `elemental`, `pure`, `recursive`, and type/attribute prefixes.

### Adapter Wrappers

`radiation.py::sunray_sw -> f2py_sunray_sw` is reported as a public signature mismatch
against raw `src/Radiation/rad_lwsw_module.F90::sunray_sw`.

This is not automatically a defect. `f2py_sunray_sw` intentionally supplies several raw
Fortran arguments from stored grid/radiation state and constants. Python exposes a
smaller adapter-facing signature, while the wrapper calls the full source routine.

Recommendation:

- For routines backed by f2py wrapper adapters, compare public Python against the f2py
  wrapper's public non-output signature.
- Separately compare the wrapper's call into the raw Fortran source routine to ensure it
  supplies the source arguments in the correct order.

### Generic Interfaces And External Libraries

Several `MISSING_FORTRAN_SOURCE_MAPPING` entries look like audit coverage gaps rather
than direct API defects:

- `grid_class.py::zt2zm`, `zm2zt`, `ddzm`, `ddzt` likely map through generic
  interfaces or wrapper-specific variants.
- `lapack_interfaces.py::*` wrap external LAPACK calls rather than normal CLUBB source
  routines.
- stats and error-code getters/setters often operate on stored state or wrapper helper
  routines rather than direct source routines.

Recommendation:

- Add explicit handling for generic interfaces and external-library wrappers.
- Classify true external wrappers separately from missing CLUBB source mappings.

## Recommended Audit Direction

Split the audit into layered checks:

1. Public Python API check:
   - Public Python args should match the intended public contract exactly.
   - For direct bindings, that contract is source Fortran `intent(in)` plus
     `intent(inout)`, in source order.
   - For adapter wrappers, that contract is the f2py wrapper's non-output args, in
     wrapper order.

2. Wrapper-to-source check:
   - Every f2py wrapper that calls source Fortran should pass source arguments in the
     exact source order.
   - Arguments supplied from stored objects/constants should be allowed here only if the
     wrapper call is explicit and order-correct.

3. Internal-only public API check:
   - Arguments such as `sclr_dim_transport`, `edsclr_dim_transport`, and
     `hydromet_dim_transport` must never appear in public Python API signatures.
   - They may appear only under the hood in wrapper/source call plumbing.

4. Missing-mapping check:
   - Report missing source mappings only after excluding G-unit sources, external
     libraries, generic-interface wrappers, and state getter/setter adapters.

## Suggested Next Fixes Before API Repair

Before using `mismatch_report` as a to-do list for API repair, improve the audit by:

1. Excluding `src/G_unit_test*` sources from contract discovery.
2. Parsing prefixed Fortran routines such as `elemental subroutine`.
3. Separating direct source bindings from adapter-wrapper bindings.
4. Adding explicit classifications for LAPACK/external wrappers and state helper
   wrappers.
5. Re-running the audit and regenerating `mismatch_report`.

After those changes, the remaining `PUBLIC_API_SIGNATURE_MISMATCH` entries should be
much closer to real API drift.
